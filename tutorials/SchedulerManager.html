This is a brief tutorial on how to configure the [SchedulerManager]{@link scheduler/SchedulerManager} class and use it submit jobs, wait for their completion and get the results back. The following is everything you need to use in order to have access to all the important information (as specified in [jobStatusInformation]{@link module:scheduler/monitors~jobStatusInformation}) of a job whose execution has ended. A complete example is featured at the end of the how-to section of the tutorial.
The last section consists of an explanation of how the module actually works.

Disclaimer: the choices made regarding the input parameters, their meaning and their behavior, the events the monitoring functions check as well as many other details all follow the specifications provided.

Using SchedulerManager

Configuring SchedulerManager

Configuration is done via a JSON file, ./input_files/input.json by default. If you want settings to be read from a different file, simply edit the path of the file where the SchedulerManager is instantiated. The parameters that can be specified are the following:
  • [maxRequestsPerSecUser]{@link scheduler/SchedulerManager#maxRequestsPerSecUser_}: maximum number of requests a user can issue in a single time unit ([requestLifespan]{@link scheduler/SchedulerManager#requestLifespan_}). Requests received by a user who has issued the maximum number of requests will not be forwarded to the SGE (unless the user submitting the request is whitelisted).
  • [maxRequestsPerSecGlobal]{@link scheduler/SchedulerManager#maxRequestsPerSecGlobal_}: maximum number of requests that can be issued by all users collectively in a single time unit ([requestLifespan]{@link scheduler/SchedulerManager#requestLifespan_}). This is a stronger constraint than [maxRequestsPerSecUser]{@link scheduler/SchedulerManager#maxRequestsPerSecUser_}. For example, if three users issued one request each in a given time unit, and [maxRequestsPerSecGlobal]{@link scheduler/SchedulerManager#maxRequestsPerSecGlobal_} was set to 3, another request by any user would be rejected (unless the user submitting the request is whitelisted).
  • [userLifespan]{@link scheduler/SchedulerManager#userLifespan}: time (in seconds) that must pass without any requests being issued by a given user, after which said user will be removed from the user history. The next time the user will submit a request, it will be as if it was the first time the user ever tried to issue one.
  • [requestLifespan]{@link scheduler/SchedulerManager#requestLifespan_}: time (in seconds) that a request issued by a user spends in the user's request history, after which it is deleted. This parameter represents the "time unit" mentioned in [maxRequestsPerSecUser]{@link scheduler/SchedulerManager#maxRequestsPerSecUser_} and [maxRequestsPerSecGlobal]{@link scheduler/SchedulerManager#maxRequestsPerSecGlobal_}.
  • [maxConcurrentJobs]{@link scheduler/SchedulerManager#maxConcurrentJobs_}: maximum number of jobs that can be handled by the SGE at the same time. Requests received by a user when the maximum number of concurrent jobs has already been reached will not be forwarded to the SGE (unless the user submitting the request is whitelisted).
  • [maxJobRunningTime]{@link scheduler/SchedulerManager#maxJobRunningTime_}: maximum time (in seconds) a [SINGLE]{@link JOB_TYPE} job submitted to the SGE can spend in the RUNNING state, after which it is terminated and removed from the job history.
  • [maxJobQueuedTime]{@link scheduler/SchedulerManager#maxJobQueuedTime_}: maximum time (in seconds) a [SINGLE]{@link JOB_TYPE} job submitted to the SGE can spend in the ON_HOLD/QUEUED state (without ever switching to RUNNING), after which it is terminated and removed from the job history.
  • [maxArrayRunningJobTime]{@link scheduler/SchedulerManager#maxArrayJobRunningTime_}: maximum time (in seconds) an [ARRAY]{@link JOB_TYPE} job submitted to the SGE can spend in the RUNNING state, after which it is terminated and removed from the job history. Note that an [ARRAY]{@link JOB_TYPE} job is considered to be RUNNING once the first task starts running.
  • [maxJobQueuedTime]{@link scheduler/SchedulerManager#maxArrayJobQueuedTime_}: maximum time (in seconds) an [ARRAY]{@link JOB_TYPE} job submitted to the SGE can spend in the ON_HOLD/QUEUED state (without ever switching to RUNNING), after which it is terminated and removed from the job history. Note that an [ARRAY]{@link JOB_TYPE} job is considered to be ON_HOLD/QUEUED as long as its first task is ON_HOLD/QUEUED itself.
  • [localListPath]{@link scheduler/SchedulerManager#localListPath_}: path to the JSON file listing the blacklisted and whitelisted users for the service this specific instance of [SchedulerManager]{@link scheduler/SchedulerManager} module is being used for. The blacklist and whitelist fields are arrays of strings. Regular expressions are supported. See ./input_files/local-lists.json for an example.
  • [globalListPath]{@link scheduler/SchedulerManager#globalListPath_}: path to the JSON file listing the blacklisted and whitelisted users for all services. Regular expressions are supported. See ./input_files/global-lists.json for an example.
    If a user is present in the blacklist or whitelist defined in this file, then the user is blacklisted or whitelisted from any and all services, regardless of their presence in the lists specific to any given service.
  • [minimumInputUpdateInterval]{@link scheduler/SchedulerManager#minimumInputUpdateInterval_}: minimum time interval (in seconds) between two consecutive reads of the input file. The input file is read whenever a user submits a request, provided the previous read happened at least minimumInputUpdateInterval seconds ago.
  • [jobPollingInterval]{@link scheduler/SchedulerManager#jobPollingInterval}: time (in seconds) dictating how often the status and several other information of a job submitted to the SGE must be checked.
  • [userPollingInterval]{@link scheduler/SchedulerManager#userPollingInterval_}: time (in seconds) dictating how often the user history must be checked, in order to detect and purge requests which have exceeded their requestLifespan or users whose userLifespan has expired.
  • [listPollingInterval]{@link scheduler/SchedulerManager#listPollingInterval_}: time (in seconds) dictating how often to read the blacklist and whitelist files specified by [localListPath]{@link scheduler/SchedulerManager#localListPath_} and [globalListPath]{@link scheduler/SchedulerManager#globalListPath_}, in order to update the blacklist and whitelist.

NOTE: a whitelisted user's requests will ALWAYS be forwarded to the SGE. Likewise, a blacklisted user's requests will ALWAYS be discarded.
NOTE: the input file is read whenever a request is received, provided the last time the file was read was at least [minimumInputUpdateInterval]{@link scheduler/SchedulerManager#minimumInputUpdateInterval_} seconds ago. Due to the non-blocking nature of this operation, it may happen that any change to the input parameters will be effective starting from the next request received by the system, and not the one that triggered the update of the input parameters: it is indeed possible that the request can be serviced faster than the file can be read and the parameters updated.

Submitting a job

An instance (named "Scheduler") of the [SchedulerManager]{@link scheduler/SchedulerManager} class as well as an instance (named "sessionManager") of [SessionManager]{@link SessionManager} are created and exported when the server is launched.
To submit a job, [handleRequest]{@link scheduler/SchedulerManager#handleRequest} must be called. This method requires a {@link requestData} object, which specifies who issued the request, at what time, and the details of the job to submit in the form of a {@link JobTemplate} object.
An example of {@link requestData} would be:

    let requestData = {
      ip: '127.0.0.1', // Just an example. Configure it properly during actual use!
      time: new Date().getTime(), // Approximate time at which the request was received.
      jobData: {
        remoteCommand: '/path/of/your/job/file.sh',
        workingDirectory: '/your/working/directory',
        jobName: 'name of your job',
      },
    };


[handleRequest]{@link scheduler/SchedulerManager#handleRequest} also requires a {@link Session} object, so the {@link Session} must be created beforehand, using the [createSession]{@link SessionManager#createSession} method, which returns a promise. For example: 

sessionManager.createSession(sessionName).then( (session) => {
  // Do something.
}, (error) => {
  // Something went wrong with the creation of the session.
});


For all possible ways to specify the properties of the jobData object, refer to the {@link JobTemplate} documentation.

It is now possible to call the [handleRequest]{@link scheduler/SchedulerManager#handleRequest} method, which returns a promise. As explained in the [handleRequest]{@link scheduler/SchedulerManager#handleRequest} documentation, the promise resolves only if a job is succesfully submitted to the SGE; the promise is otherwise rejected. To attempt to submit the job and check if the submission was successful, one can do the following: 

Sec.handleRequest(requestData, session).then( (status) => {
  // Do something with the result.
}, (error) => {
  // The request could not be accepted for some reason.
});


The status object which the promise resolves into in case of success is a {@link requestOutcome} object holding several useful information, the most important of which is the job ID (stored in the jobId field) assigned by the SGE to the job that has just been submitted.
In case of rejection, the object returned is also of the {@link requestOutcome} kind. The reason for the rejection can be found in the "errors" field of the object. NOTE: a single call of [handleRequest]{@link scheduler/SchedulerManager#handleRequest} handles the submission of ONE job (be it [SINGLE]{@link JOB_TYPE} or [ARRAY]{@link JOB_TYPE}). If you want to submit another job, then [handleRequest]{@link scheduler/SchedulerManager#handleRequest} must be called again.

Examining the status and other relevant information of a completed job

To monitor a submitted job, it is sufficient to call the [getJobResult]{@link scheduler/SchedulerManager#getJobResult} method, which requires the id of the job to monitor and the session the job was launched in. This method returns a promise. As explained in the documentation, said promise resolves when the job that is being monitored enters the COMPLETED or ERROR state, while it is rejected when an error of the monitoring process occurs: if a job is in ERROR status, the promise resolves anyway.
In case of success, the promise resolves into a [jobStatusInformation]{@link module:scheduler/monitors~jobStatusInformation} object, containing several information about the now finished job.
In case of rejection, the promise returns an object whose "errors" field contains information regarding the error that occurred.
For example, to monitor the status of a job whose id is stored in a jobId variable, one would do:

Sec.getJobResult(status.jobData.jobId, session).then( (status) => {
  // The job terminated its execution in an expected manner.
}, (error) => {
  // Something went wrong while monitoring the job.
})

let requestData = {
      ip: '127.0.0.1', // Just an example. Configure it properly during actual use!
      time: new Date().getTime(), // Approximate time at which the request was received.
      jobData: {
        remoteCommand: '/path/of/your/job/file.sh',
        workingDirectory: '/your/working/directory',
        jobName: 'name of your job',
      },
    };

    sessionManager.createSession('yourSession')
        .then(
            (session) => {
              Sec.handleRequest(requestData, session)
                  .then(
                      (status) => {
                        console.log('Request outcome: ' + status.description);
                        Sec.getJobResult(status.jobData.jobId, session)
                            .then(
                                (status) => {
                                  console.log(
                                      'Job ' + status.jobId + ' of session ' +
                                      status.sessionName + ' status: ' +
                                      status.mainStatus + '-' +
                                      status.subStatus + ', exitCode: ' +
                                      status.exitStatus + ', failed: ' +
                                      status.failed + ', errors: ' +
                                      status.errors + ', description: ' +
                                      status.description);
                                },
                                (error) => {
                                  console.log('Error: ' + error.errors);
                                });
                      },
                      (error) => { console.log('Error: ' + error.errors); });
            },
            (error) => {
              console.log(
                  'Could not create session ' + sessionName + ': ' + error);
            });

Software behavior

How a request is handled by the module

The moment a request is submitted via [handleRequest]{@link scheduler/SchedulerManager#handleRequest}, the [verifyRequest_]{@link scheduler/SchedulerManager#verifyRequest_} method checks if the user is whitelisted ([isWhitelisted_]{@link scheduler/SchedulerManager#isWhitelisted_}), in which case the request can be serviced.
If the user is not whitelisted, additional checks are performed in order to verify that no constraints are violated by the request. These constraints are, in order of importance:
  • The user must not be blacklisted ([isBlacklisted_]{@link scheduler/SchedulerManager#isBlacklisted_}).
  • The current number of jobs currently being handled by the SGE must be less than [maxConcurrentJobs]{@link scheduler/SchedulerManager#maxConcurrentJobs_}.
    If multiple requests have been received at the same time (i.e. an epsilon milliseconds apart) and more than one is still being examined ([pendingJobsCounter]{@link scheduler/SchedulerManager#pendingJobsCounter_}) as the request in question itself is being examined, they are considered as if they were already submitted to the SGE. This is necessary to ensure that the [maxConcurrentJobs]{@link scheduler/SchedulerManager#maxConcurrentJobs_} constraint is not violated when multiple requests happen at the same time.
  • The number of requests (not including the one being examined) made by all users in the last [requestLifespan]{@link scheduler/SchedulerManager#requestLifespan_} seconds must be less or equal than [maxRequestsPerSecGlobal]{@link scheduler/SchedulerManager#maxRequestsPerSecGlobal_}.
  • The number of requests (not including the one being examined) made by the user in the last [requestLifespan]{@link scheduler/SchedulerManager#requestLifespan_} seconds must be less or equal than [maxRequestsPerSecUser]{@link scheduler/SchedulerManager#maxRequestsPerSecUser_}.

If all these checks are passed, the [handleJobSubmission]{@link scheduler/SchedulerManager#handleJobSubmission} attempts to submit the job to the SGE. The outcome of the submission, whether it succeeded or failed, is returned in a {@link requestOutcome} object.

How the module monitors a submitted job

A job can be monitored via the [getJobResult]{@link scheduler/SchedulerManager#getJobResult} method, which keeps calling [monitorJob]{@link scheduler/monitors#monitorJob} until the job execution ends (whether successfully or because of an error in the job file being executed or a timeout being exceeded) or an error in the monitoring process occurs.
[monitorJob]{@link scheduler/monitors#monitorJob} delegates the actual monitoring to [pollJob]{@link scheduler/monitors#pollJob}, which is called every [jobPollingInterval]{@link scheduler/SchedulerManager#jobPollingInterval} seconds. [pollJob]{@link scheduler/monitors#pollJob} uses the functions [monitorSingleJob]{@link scheduler/monitors#monitorSingleJob} or [monitorArrayJob]{@link scheduler/monitors#monitorArrayJob} (according to the job's [JOB_TYPE]{@link JOB_TYPE}) to check the status of the specified job to confirm if any meaningful events occurred.
The main events being monitored are the time the job spends in the ON_HOLD/QUEUED state, the time spent in the RUNNING state, or if the job enters the ERROR state.
The following conventions were established (as per specifications) for [ARRAY]{@link JOB_TYPE} jobs:

  • The job as a whole is ON_HOLD/QUEUED as long as its first task is. When said task moves to the RUNNING state, the job is considered to be RUNNING.
  • The RUNNING time of the job consists of the cumulative RUNNING time of each task of the job. For example, if the job consists of 3 tasks and each task runs for 5 seconds, the total RUNNING time is approximately 15 seconds (5 per task for a total of 15).
  • If at least one task is in ERROR, then the job as a whole is.
